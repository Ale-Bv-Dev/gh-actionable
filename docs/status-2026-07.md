## Filtri v1 attivi (da `docs/specs/mvp-v1.md`)

**Hard filter** (esclusione, tutti richiesti):
- issue aperta
- ha label `good first issue` o `help wanted`
- nessun assignee
- nessuna PR aperta collegata
- nessuna label negativa (`wontfix`, `duplicate`, `invalid`, `stale`, `status: stale`)
- attività umana reale negli ultimi 90 giorni

**Anti-ghost warning** (informativi, non escludono):
- ultimo commento da bot
- keyword conservative nell'ultimo commento rilevante (`won't fix`, `duplicate of`, `closing as`)

**Soft signal** (positivi, per issue già incluse):
- presenza `CONTRIBUTING.md`
- assenza di ghost warning
- commento umano recente
- corpo issue strutturato/descrittivo

Ordinamento: `updated_at` decrescente. Cache: JSON con TTL, per-repo, redatta (niente body/commenti in cache dalla fix F-02).

## Motivo della pausa

v1 è stato costruito, testato (183 test) e sottoposto a smoke-test su repo reali (es. `ethereum/ethereum-org-website`). Durante gli smoke-test sono emersi **falsi positivi** non coperti dai filtri hard attuali:

- F-04 ha corretto un caso diretto (`Status: Stale` non normalizzato in `NEGATIVE_LABELS`), risolvibile con un filtro esatto.
- Restano categorie di falso positivo **non gestibili con un filtro secco**, perché richiedono segnali più sfumati: claim informale in un commento senza assignment formale, PR aperta che cita l'issue senza link GitHub nativo, label "help wanted" ormai vecchia senza attività maintainer, maintainer che risponde solo "is this still open?" senza intenzione di seguire.

Questi casi non sono quantificati: non esiste ancora un dataset etichettato che misuri quanto pesano sulla precision reale. Da qui l'obiettivo v2: **misurare prima di correggere** (FASE 1 — eval set), poi intervenire filtro per filtro sui dati (FASE 2), invece di aggiungere euristiche non verificate.

## Obiettivo v2

Alzare la precision del tool tramite un ciclo eval-driven:
1. Tassonomia dei falsi positivi + dataset etichettato manualmente (30–50 candidati)
2. Precision baseline misurata
3. Filtri v2 aggiunti uno alla volta, ciascuno validato sul delta di precision misurato, non per intuizione
4. Target dichiarato: precision ≥ 70% sull'eval set (rivedibile al gate umano)

Nessuna modifica ai filtri v1 è stata fatta in questa fase (FASE 0 è sola lettura).
